# GuanZai AI 项目保姆与可组合能力设计

日期：2026-07-15  
状态：已完成对话设计，等待书面审阅

## 1. 背景

GuanZai `v0.3.0-alpha.1` 当前是一个确定性的任务规划核心。它可以判断任务的编排价值、操作影响、审计需要和建议模型，生成标记为 `execution = planned` 的清单，但不会启动工作者、追踪执行、收集验证结果、管理合并或部署。

现有定位主要回答“是否值得委派、由谁处理、使用什么成本层级”。这不足以满足非程序员长期维护项目的需要。用户真正需要的是一个统一入口，帮助其：

1. 在行动前把项目目标、对象、假设和成功标准想清楚；
2. 根据任务状态选择研究、实验、设计、执行、诊断或停止；
3. 在需要代码时保护正式版本，并收集可信的验证证据；
4. 在重要变化前用普通语言解释前因后果；
5. 把最终合并决定保留给用户；
6. 用最小充分上下文恢复工作，避免反复加载完整历史；
7. 让项目经验能够验证、版本化、更新和退休。

本设计将 GuanZai 从“成本感知的模型路由器”扩展为“AI 项目保姆”，同时保留其小型、可检查和不夸大执行状态的基础。

## 2. 产品定位

### 2.1 用户可见定位

> **GuanZai 是你的 AI 项目保姆。它陪你把事情想清楚，选择合适的能力完成工作，用证据检查结果，并在重要变化发生前帮助你真正看懂和决定。**

“保姆”表示照看、提醒、解释、留痕和保护，不表示替用户拥有项目或替用户接受重大风险。

### 2.2 服务范围

GuanZai 管理的是项目生命周期，不只管理程序代码。项目产物可以是：

- 产品与服务；
- 研究、文章或报告；
- 数据分析和业务规则；
- Agent Skill 或工作流；
- 软件、网站和自动化系统；
- 尚未确定是否值得投入的想法。

代码版本控制、测试、合并和回滚是安全交付能力，只在项目需要代码、配置或可变数据时启用。

### 2.3 用户与 GuanZai 的责任边界

用户负责：

- 提出目标、背景和真实偏好；
- 纠正 GuanZai 对项目的误解；
- 理解剩余影响与风险；
- 决定是否合并正式版本；
- 对上线、删除、付款、授权等高影响操作单独授权。

GuanZai 负责：

- 判断项目当前状态和信息充分程度；
- 选择合适的思考、研究、执行和验证能力；
- 保护稳定成果，保存恢复点；
- 区分计划、执行、验证与批准；
- 用普通语言说明下一步的前因后果；
- 只保留经过验证、仍适用的经验。

## 3. 核心原则

### 3.1 软内核，硬边界

GuanZai 管结果和证据，不强迫所有任务执行相同的思考步骤。

下列内容是可调整的软建议：

- 是否写完整方案；
- 是否使用多个 Agent；
- 是否使用测试驱动开发；
- 是否需要独立审查；
- 是否创建独立工作区；
- 使用哪个模型和推理强度；
- 任务拆成多少步；
- 文档写到什么粒度。

执行者可以省略、合并或替换软步骤，但必须说明省略原因，以及风险如何得到控制。

下列内容是首版不可突破的硬边界：

1. 未经用户批准，不得合并正式版本；
2. 合并批准不等于上线、删除、付款、授权或其他高影响操作的批准；
3. 不得泄露凭据、秘密、个人隐私或受保护数据；
4. 非轻量变更必须存在可用的恢复点；
5. 不得把“测试通过”表述为“绝对没有错误”；
6. 不确定性、未验证范围和失败必须明确披露；
7. 推荐、计划、已执行、已验证和已批准必须使用不同状态；
8. 合并前必须生成标准合并决策卡；
9. 决策卡生成后若批准对象发生变化，原批准失效；
10. 无法用普通语言解释变更时，不得请求用户批准合并。

### 3.2 一站式不等于一体化

用户只需要进入 GuanZai，但 GuanZai 不复制或吞并所有专业能力。专业能力保持独立仓库、独立版本和独立测试，通过稳定的能力契约组合。

### 3.3 最轻量但足够安全

每次任务选择能够控制实际风险的最轻流程。机械工作量、长文本或大量文件本身不构成多 Agent 的理由。只有认知复杂性、独立性价值或行动后果足以覆盖编排成本时才委派。

### 3.4 证明结果，不规定思维表演

GuanZai 不要求模型展示固定的思维步骤。它要求可检查的外部产物：项目定义、证据、差异、测试结果、恢复点、决策卡和状态记录。

### 3.5 机器保存事实，模型完成判断，人批准后果

三类职责必须分开：

- **脚本和确定性工具**：记录 Git、文件、测试、任务状态、运行时间和成本等可观察事实；
- **模型和专业能力**：理解目标、挑战假设、选择路径、解释影响和判断证据是否足够；
- **用户**：纠正真实意图，接受剩余风险，批准合并和其他高影响操作。

“脚本完成”只说明机械条件成立，不自动等于需求正确或真实效果达成。任何完成判断都必须区分：

1. `mechanical`：命令、文件、测试等机械条件；
2. `semantic`：成果符合已经确认的需求和验收合同；
3. `outcome`：真实用户或业务效果达到目标。

### 3.6 上下文按需展开，证据始终可追溯

新会话和工作者默认只接收完成任务所需的最小上下文。摘要必须指向原始证据；过滤和截断必须显式标记；发现缺口时允许渐进读取。最小上下文不是越少越好，而是没有无关历史、又不缺少完成任务所需的约束和证据。

完整设计规格用于开发、审计和演进，不作为每次运行的默认提示。运行时只加载短核心原则、当前状态和本任务需要的能力契约；详细协议、历史决定和测试说明通过索引按需读取。设计深度不能转化为持续上下文税。

### 3.7 苏格拉底之门：方法柔软，本质必须确认

对于方向模糊、投入较大或后果重要的项目，GuanZai 在执行前开启“苏格拉底之门”，与用户确认双方理解的是同一个本质问题。

苏格拉底提问、第一性原理和费曼学习法是可替换的软方法：

- **一问**：用苏格拉底式问题揭开未经验证的前提、反例和真正会改变决定的信息；
- **一拆**：用第一性原理区分事实、真实约束、继承做法和已经偷渡进需求的解决方案；
- **一讲**：由 AI 用费曼式普通语言复述问题、对象、目标和成功表现，请用户纠正，而不是要求用户参加考试。

触发后的结果是条件性硬门槛：核心问题、目标对象、成功证据或关键边界仍可能改变方案时，项目不得进入 `ready`。方法可以更换、合并或省略，但不得把关键假设伪装成事实，也不得在双方尚未对齐时开始大规模执行。

## 4. 总体架构

```text
用户想法、问题或项目
          |
          v
GuanZai 入口：理解处境、权限与项目状态
          |
          v
是否需要苏格拉底之门？
     | 否                    | 是
     |                       v
     |                苏格拉底之门
     |                       |
     |                       +--> 深层方向未可信 --> Foolish Think（可选依赖）
     |                       |
     |                       v
     |                  项目本质卡
     |                       |
     +-----------------------+
                             v
                  经确认的项目定义与验证路线图
                             |
                             v
                    形成任务与验收契约
                             |
                             v
                   最小上下文包与能力路由
                             |
                             v
                   研究 / 设计 / 执行 / 诊断
                             |
                             v
                      证据与可恢复成果
                             |
                             v
                   标准决策卡与用户批准节点
                             |
                             v
                 合并；其他高影响操作另行授权
```

架构分为五层：

1. **治理层**：GuanZai 判断状态、风险、所需能力、授权和证据；
2. **状态与上下文层**：维护机器状态、短视图、决定、证据索引和渐进读取；
3. **能力层**：Foolish Think、研究、设计、开发、测试、审计等独立能力；
4. **执行层**：Codex、WorkBuddy、其他 Agent、CLI 和人工执行者；
5. **验证与授权层**：把机械事实连接到语义验收、真实效果、决策卡和用户批准。

新增能力不能改变硬边界的含义。提供商、模型和 Skill 属于能力或执行层，不拥有合并授权。

### 4.1 层间接口

| 上游 | 下游 | 最小交接物 | 拒绝接收的情况 |
| --- | --- | --- | --- |
| 用户入口 | 苏格拉底之门 | 背景、目标、已知事实、约束、待决问题 | 把功能名称直接当成真实问题 |
| 苏格拉底之门 | 项目思考或治理计划 | 项目本质卡、确认版本、仍未验证的关键假设 | 用户尚未确认核心问题，或 AI 无法通俗复述 |
| 项目思考 | 治理计划 | `R# -> O# -> S#` 或等价的项目定义与验证路线图 | 没有验收证据、失败成本或停止条件 |
| 治理计划 | 执行能力 | 任务 ID、目标、非目标、授权边界、相关决定、输入路径、输出契约、验收标准 | 只给一句模糊命令或倾倒完整历史 |
| 执行能力 | 验证层 | 不可变产物、机器结果、原始证据路径、未验证范围 | 只有“完成了”的自报结论 |
| 验证层 | 决策卡 | 机械、语义和效果证据，恢复方式，批准对象指纹 | 测试与需求脱节或批准对象可变 |
| 决策卡 | 用户 | 一屏左右的普通语言影响说明和明确建议 | 只有技术术语、CI 颜色或 Yes/No |
| 已确认结果 | 项目记忆 | 带范围、来源、证据、版本和复查日期的经验候选 | 单次偶然现象直接晋升永久规则 |

每个接口都必须允许下游因信息不足而拒绝接收，并返回缺失字段；不得靠下游猜测补齐关键授权或成功标准。

### 4.2 可替换性

能力通过契约连接，不通过模型名称连接。替换 Foolish Think、Codex、WorkBuddy、测试工具或存储实现时，只要输入输出契约和硬边界保持不变，治理层不应重写。任何新增适配器必须证明：移除它以后 GuanZai 核心仍然可以诚实降级运行。

## 5. Context Diet 与状态架构

Context Diet 是横跨所有阶段的基础设施，不是 GuanZai 的产品身份。它的目标是减少无关历史、重复总结和工具噪声，同时不牺牲事实完整性、恢复能力和审计性。

### 5.1 两类状态，单一真相源

```text
.guanzai/                       # 私有、项目本地、默认不提交
├── runtime/
│   ├── state.json             # 工作流账本和外部证据观察索引
│   └── STATE.md               # 从事实和确认注释生成的短视图
├── evidence/
│   ├── tests/
│   └── tool-output/
└── archive/                   # 已结束运行，默认不加载

docs/guanzai/                  # 适合共享且经过确认的长期项目知识
├── project-brief.md           # 当前有效项目定义和按需读取索引
├── essence/
│   ├── E-001-*.md             # 已确认或实验性的项目本质卡版本
│   └── E-002-*.md
└── decisions/
    ├── D-001-*.md
    └── D-002-*.md
```

- Git、CI、文件系统和提供商返回是各自领域的原始事实源；
- `state.json` 是 GuanZai 工作流状态与证据索引的权威账本，保存外部事实的观察值、来源和观察时间，不取代外部事实源；
- `STATE.md` 是面向人和模型的可丢弃视图，尽量由脚本渲染，不得成为第二份独立事实源；
- 模型只在重要决定、阶段切换、中断或阻碍发生时补充少量语义注释；
- 经过确认、需要跨设备或跨 Agent 延续的项目定义和决定进入版本化项目知识；
- `project-brief.md` 指向当前有效的本质卡、验证路线图和关键决定，不复制它们的完整内容；
- 私密决定可以留在私有存储，但必须明确它不会随仓库同步；
- 多个并行任务使用独立任务 ID 和独立决定文件，避免争写一个不断增长的 `decisions.md`。

所有机器状态写入必须采用 Schema 版本、原子替换和锁或等价并发保护。中断后应能识别最后一个完整状态，而不是读取半写文件。用于验证、决策卡或批准前，必须刷新会变化的外部事实；观察超过适用时效或来源已经变化时，相应账本字段标记为 `stale`。

状态刷新由事件触发，不在每轮对话后调用模型重写：

- 任务开始或恢复；
- 文件、分支、提交或外部系统发生修改；
- 一组验证完成或失败；
- 项目阶段、授权边界或重要决定变化；
- 即将中断、压缩上下文或交接；
- 生成决策卡和执行批准操作之前。

没有状态变化的普通对话不产生新的 handoff。

### 5.2 短视图的信息预算

`STATE.md` 默认以约 1–2 KB 为目标，而不是不可突破的长度限制。它只保留：

- 当前目标和非目标；
- 当前阶段和批准边界；
- 已完成事项；
- 当前阻碍和未验证范围；
- 待决问题和下一步；
- 相关决定、证据和恢复点链接；
- 生成时间与来源状态版本。

超出预算时优先归档已经结束的阶段，不压缩仍然有效的风险、授权或恢复信息。新会话默认只读短视图；需要追溯时再按链接读取决定和档案。

### 5.3 共享桥

跨 Agent 的共享桥只保留当前有效能力、最短路由说明和按需读取入口。提供商协议、历史决定、长期能力说明和已结束月份进入独立文件。共享桥是索引，不是所有规则和历史的副本。

全局能力信息与项目决定必须分开：前者描述宿主能够做什么，后者描述本项目为什么这样做。项目恢复不能依赖某台机器上未同步的全局桥。

### 5.4 工具输出预算与证据保全

工具输出默认先过滤、统计、脱敏，再进入模型：

- 文件发现优先使用文件列表和精确模式；
- 文本读取优先读取目标段落，并设置输出预算；
- JSON 只选择任务所需字段；
- 日志先提取退出码、错误数、关键警告和相关上下文；
- 二进制、缓存、构建产物和历史索引默认排除；
- 原始输出保存为证据文件，记录来源、时间、退出码、大小和可选哈希；
- 摘要必须说明原始输出总量、本次展示范围以及是否截断；
- 发现异常、冲突或证据不足时，允许自动扩大相关上下文。

过滤不能静默隐藏内容。无法安全脱敏的原始输出不得发送给外部模型，也不得不加说明地持久化。

### 5.5 最小任务包

工作者不继承完整对话。任务包至少包含：

- 不可变任务 ID；
- 目标和非目标；
- 当前项目阶段、风险与授权边界；
- 相关决定 ID，而不是全部决定正文；
- 输入文件、证据和按需读取路径；
- 输出格式和验收证据；
- 允许修改的范围；
- 失败、冲突和信息缺失的返回协议。

工作者发现任务包不足时应请求指定上下文，不得为了保持低 token 而猜测关键事实。

## 6. Foolish Think 的组合方式

### 6.1 架构决定

Foolish Think 与 GuanZai 保持两个仓库、两个 Skill 和独立发布。GuanZai 不复制 Foolish Think 的完整指令，只保留：

- 能力标识与用途；
- 语义触发条件；
- 输入输出契约；
- 支持版本范围；
- 官方仓库和安装来源；
- 能力检测、缺失提示与降级规则。

Foolish Think 官方仓库：<https://github.com/Dovesoup/foolish-think>

### 6.2 语义路由

当项目定义、核心假设、可行性、方向或实际结果尚不可信时，GuanZai 推荐项目思考能力：

- `Seed`：只有需求或早期想法；
- `Stress-test`：已有方案但尚未发生重大失败；
- `Diagnose`：执行结果与预期存在重要差距。

下列情况默认跳过 Foolish Think：

- 已批准方案下的直接执行；
- 独立且可复现的技术故障；
- 普通知识问答；
- 不需要战略审视的轻量修改；
- 用户明确要求不做方向挑战，且行动不触发硬安全边界。

当另一个 brainstorming 流程已在运行时，不重复启动第二套需求发现对话。保留当前对话和批准节奏，同时应用 Foolish Think 的假设审计、失败预演、可追溯性和证伪标准。

### 6.3 交接契约

Foolish Think 的项目级输出向 GuanZai 提供三类可追溯对象：

```text
R#：事实、假设、风险或根因发现
  -> O#：针对具体发现的方向调整
      -> S#：能够验证或证伪方向的行动单元
```

`S#` 成为 GuanZai 后续计划的验收输入，至少包含：

- 目标和交付物；
- 输入与输出契约；
- 客观验收证据；
- 最小证伪实验；
- 失败成本与可逆性；
- 依赖和下游消费者；
- `do now`、`experiment first` 或 `restart` 决定。

这样，项目最初的关键假设能够追踪到最终验收和合并决策，不允许执行者在末端自行改写成功标准。

### 6.4 安装与依赖管理

GuanZai 核心必须能够独立安装和运行。Foolish Think 是推荐但可选的能力依赖。

未来的 `guanzai setup` 提供一站式安装：

1. 显示核心和推荐能力；
2. 解释每项能力的用途、来源、版本和数据边界；
3. 获得用户明确同意后，从官方仓库获取固定版本；
4. 校验版本、发布来源和文件完整性；
5. 安装到当前 Agent 平台的个人 Skill 目录；
6. 检查文件是否存在，并区分“已安装”和“当前会话已加载”；
7. 平台需要重启时明确提示；
8. 记录已安装版本和兼容性结果。

不得静默下载 `latest`，不得自动执行未经固定和校验的远程安装脚本。

未来的 `guanzai doctor` 对每项能力报告：

- `missing`：未安装；
- `installed`：文件存在，但当前会话是否加载未知；
- `available`：当前宿主明确暴露该能力；
- `incompatible`：版本不在支持范围；
- `unverified`：只能检测文件，无法证明运行可用。

当任务需要 Foolish Think 但能力不可用时，GuanZai 必须：

1. 说明为什么建议使用；
2. 说明当前缺失状态；
3. 提供经用户确认的一键安装；
4. 允许用户拒绝；
5. 拒绝后使用基础澄清能力继续，并披露未执行完整假设审查；
6. 不得模仿或声称已经运行完整 Foolish Think。

## 7. 项目生命周期

### 7.1 入口状态判断

每项工作先归入一个主状态：

- `idea`：想法或需求尚未形成可信项目；
- `proposal`：已有方案，尚待压力测试或批准；
- `ready`：方向和验收条件已确认，可以执行；
- `active`：项目正在执行；
- `blocked`：执行受阻，但根因未明；
- `degraded`：已有成果，但实际效果不符合预期；
- `ready_for_decision`：证据满足当前验收合同，批准对象已冻结并等待用户决定；
- `complete`：所需决定和操作已经完成，成果被正式接受；
- `stopped`：决定停止投入，并记录原因与保留成果。

状态判断可以由强模型进行语义分析。确定性规则只作为可解释的保底，不应假装具有完整语义理解。

### 7.2 苏格拉底之门

#### 触发条件

满足任一条件时进入：

- 真实问题、目标用户或成功标准不清楚；
- 用户表达的是一个功能或方案，但背后的问题尚未确认；
- 某个未决答案会明显改变成本、架构、风险或验收方式；
- 项目将产生较大投入、长期耦合或难以撤销的后果；
- 用户明确希望共同深入理解项目本质。

下列情况默认跳过：

- 已经批准的规格下进行直接执行；
- 目标和验收明确的低风险、易撤销修改；
- 独立且可复现的技术故障；
- 普通知识问答、事实读取和机械整理。

#### 对话纪律

- 每次只问一个最高信息价值问题；
- 提问前用一句话说明这个答案会影响什么，让用户知道为什么要问；
- 只有答案可能改变方向、风险、边界或验收时才提问；
- 能从项目文件、代码、数据或现有证据发现的答案，不反问用户；
- 问题不能把 AI 偏好的方案暗藏在问题里；
- 尊重“我不知道”，必要时提供通俗例子或互斥选项帮助表达，不把对话变成考试；
- 使用用户熟悉的语言，不以术语、反诘或居高临下的语气制造压力；
- 不追求消灭所有未知，只处理会影响当前决定的未知；
- 信息充分后立即停止提问，不把深入思考变成拖延；
- 用户可以纠正、暂缓或拒绝某项假设，但硬安全边界不因此失效。

第一性原理不是默认否定历史经验。法律、物理限制、真实用户行为和经过证据支持的组织约束应被保留；只有未经验证却被当成必然的继承做法才进入挑战清单。

#### 项目本质卡

触发后的最小交付物是一个可版本化的项目本质卡：

```text
我们真正要解决的问题：
受到影响的人及其当前做法：
为什么现有方式不够：
已经确认的事实：
仍未验证的关键假设：
真实且不可忽略的约束：
这次明确不解决什么：
最小成功证据：
最低成本的证伪方式：
AI 的一句话通俗复述：
用户确认状态与版本：
```

字段按适用性填写，不为了填表制造内容。轻量澄清可以只有一句本质复述、一个关键假设和用户确认。

项目本质卡具有不可变版本。核心问题、对象、成功证据或非目标发生实质变化时：

1. 本质卡生成新版本；
2. 依赖旧版本的方案、任务包和验收合同标记为 `stale`；
3. 重新判断是否需要 Foolish Think；
4. 只重做受影响的下游工作，不机械重启整个项目。

#### 与 Foolish Think 的关系

苏格拉底之门是 GuanZai 的入口治理条件，不是第三个独立 Skill。它回答“我们是否理解了同一个问题”。Foolish Think 是可选的深层项目思考能力，回答“项目的假设、方向和证据是否经得起挑战”。

当 Foolish Think 被调用时，其 Stage 1 同时承担苏格拉底之门的深层澄清，不再运行第二套重复问答。输出的 `R# -> O# -> S#` 与项目本质卡共享事实、假设和确认版本。

#### 状态

`essence_status` 独立记录：

- `not_required`：本次任务无需开启；
- `needs_clarification`：存在会改变决定的关键未知；
- `awaiting_confirmation`：AI 已通俗复述，等待用户纠正或确认；
- `provisional_experiment`：本质尚未确认，只批准了低成本证伪实验；
- `confirmed`：用户确认指定版本；
- `stale`：项目本质发生变化，旧确认失效。

只有 `not_required` 或 `confirmed` 才允许进入 `ready`。`provisional_experiment` 只能执行已确认边界内的实验，实验结果返回苏格拉底之门或 Foolish Think 更新本质卡。

### 7.3 风险等级

| 等级 | 典型任务 | 默认路径 |
| --- | --- | --- |
| 轻量 | 文案、格式、局部样式、可轻易撤销的整理 | 直接执行、结果检查、简短说明 |
| 普通 | 新功能、数据处理、规则或工作流修改 | 简短方案、隔离执行、验证、标准决策卡 |
| 高风险 | 生产数据、权限、安全、财务、架构迁移、不可逆操作 | 项目/方案审查、独立验证、恢复演练、标准决策卡和分离授权 |

风险取决于行动和影响，不取决于敏感词本身。只读安全说明不等同于安全系统变更；一行生产计费公式修改仍是高风险任务。

### 7.4 代码与可变项目的安全交付

普通及高风险代码变更默认：

```text
已确认的正式版本
  -> 独立分支或工作区
      -> 小步修改与可识别恢复点
          -> 自动测试和必要的真实效果检查
              -> 独立验证（按风险）
                  -> 标准合并决策卡
                      -> 等待用户批准合并
```

用户选择的默认授权边界是：**自动工作只能运行到“等待确认合并”为止。**

以下操作不包含在合并批准中，必须根据影响单独请求授权：

- 部署到生产环境；
- 数据删除或不可逆迁移；
- 权限、密钥和访问控制变更；
- 付款、转账或计费；
- 清理仍承担恢复用途的分支、工作区、备份或临时数据库。

若项目没有 Git，GuanZai 应优先创建可恢复副本或快照，并明确说明保护程度低于完整版本控制，不得使用“合并”掩盖普通文件覆盖。

## 8. 标准合并决策卡

决策卡默认控制在一屏左右，以普通语言回答：用户批准后世界会发生什么变化。

```text
结论：建议合并 / 建议暂缓 / 需要你先检查

原本的问题：
这次做了什么：
合并后的实际变化：
为什么采用这个方案：
验证了哪些情况：
没有验证或仍不确定的地方：
可能影响谁：
出现问题如何恢复：
AI 的明确建议：
你现在批准的具体事项：
批准对象指纹：
```

### 8.1 生成规则

- 不得只报告测试数量、覆盖率或 CI 颜色；
- 必须解释验证证据证明了什么、没有证明什么；
- 必须区分程序正确性、需求正确性和真实使用效果；
- 必须披露未运行、失败、跳过或不适用的验证；
- 必须给出明确建议和依据，不能把判断完全推回给用户；
- 技术附件可以保留，但不得替代普通语言摘要。

### 8.2 批准对象

决策卡必须绑定不可歧义的批准对象，例如 Git 提交 SHA、不可变构建标识或文件快照哈希。

生成决策卡后，只要批准对象发生变化：

1. 原决策卡标记为 `stale`；
2. 原批准自动失效；
3. 重新验证变化部分及其影响；
4. 生成新决策卡；
5. 再次等待用户批准。

## 9. 状态与证据模型

### 9.1 状态必须真实

建议使用下列工作状态：

- `recommended`：建议使用某能力；
- `planned`：已生成计划；
- `available`：宿主暴露能力；
- `started`：执行已经开始；
- `completed`：执行者报告完成；
- `verified`：验收证据经过检查；
- `awaiting_approval`：批准对象已冻结并生成决策卡；
- `approved`：用户批准指定对象；
- `merged`：指定对象已经进入正式版本；
- `failed`：未完成或证据不满足；
- `stale`：对象变化导致计划、证据或批准失效；
- `stopped`：基于明确决定停止投入。

状态只能由相应证据推进。例如，存在 CLI 不代表 `available`，生成命令不代表 `started`，执行者自报完成不代表 `verified`。

运行状态、项目状态和验收状态不能塞进一个模糊的 `status` 字段：

- **运行状态**回答工作者是否开始、完成或失败；
- **项目状态**回答项目处于想法、方案、执行、受阻、退化或停止阶段；
- **验收状态**回答机械、语义和效果证据是否满足；
- **授权状态**回答用户是否批准了哪个不可变对象和哪一种操作。

各状态轴独立推进。工作者可以 `completed`，但语义验收仍为 `unverified`；代码可以 `approved` 合并，但部署授权仍为 `not_requested`。

### 9.2 计划清单扩展

后续清单应增加：

```json
{
  "project_state": "proposal",
  "essence": {
    "status": "awaiting_confirmation",
    "card_version": "E-001",
    "confirmed_by_user": false
  },
  "risk_level": "ordinary",
  "hard_boundaries": [],
  "soft_playbook": [],
  "required_evidence": [],
  "context": {
    "state_version": 1,
    "summary_path": ".guanzai/runtime/STATE.md",
    "truncated": false
  },
  "approval_policy": {
    "merge": "user_required",
    "deployment": "separate_authorization"
  },
  "required_capabilities": [
    {
      "name": "project-thinking",
      "provider": "foolish-think",
      "status": "missing",
      "required": false,
      "reason": "项目方向和关键假设尚未得到验证"
    }
  ],
  "decision_card_required": true
}
```

字段是机器契约，面向用户的输出仍需转换为普通语言。

### 9.3 模型路由的可观察性

模型策略不能把建议、请求和运行事实混为一谈。每个工作项至少区分：

```text
recommended_model        # 策略建议
requested_model          # 发送给宿主或 CLI 的请求
observed_runtime_model   # 宿主实际返回或可观察到的模型；不可见时为 null
runtime_model_evidence   # 原始返回字段、命令回显或提供商记录
runtime_model_confidence # verified / reported / inferred / unknown
```

自然语言中的“换成某模型”只能产生意图，不能直接写入 `observed_runtime_model`。原生接口无法逐 Agent 指定模型时，必须报告宿主限制；确需强制模型时只能使用真实支持参数并能保留运行证据的适配器。

治理策略优先描述任务需要的能力、风险和可验证性，不把模型名称固化成角色。模型目录、价格、上下文限制和可用性通过版本化能力注册表更新。

### 9.4 证据链

每项完成声明应能追溯：

```text
目标/假设
  -> 验收合同
      -> 执行产物
          -> 机器证据
              -> 语义验收
                  -> 真实效果（适用时）
                      -> 决策卡
                          -> 用户批准对象
```

原始证据保存在文件中，模型摘要记录路径和范围。证据后来变化、丢失或无法读取时，依赖该证据的验证和批准状态必须降级或标记为 `stale`。

## 10. 失败与降级处理

### 10.1 信息不足

如果缺失信息可能改变项目方向、根因判断或第一项不可逆决定，停止扩展方案，每次只询问一个最高信息价值问题。

### 10.2 能力缺失

说明缺少什么、为什么需要、如何安装、拒绝后的影响。降级继续时必须标记哪些检查未执行。

### 10.3 无可靠验证方法

不得用“看起来正常”替代证据。提供最低成本的替代验证，或将结论改为“需要你先检查”。

### 10.4 工作区不安全

遇到用户未提交的修改、来源不明的文件、无法隔离的正式环境或没有恢复点时，不覆盖、不清理；先保护现状，再说明阻碍。

### 10.5 验证失败

保持正式版本不变。修复后重新运行受影响验证；连续失败应重新检查方案假设，而不是无限重复同一路径。

### 10.6 批准过期

批准对象变化后不沿用旧的 Yes。状态退回 `stale`，重新生成证据和决策卡。

### 10.7 无法恢复

在第一次难以撤销的操作前停止，说明最坏损失和替代方案，获得单独授权；不能把它包装为普通合并。

### 10.8 状态损坏或并发冲突

无法解析 `state.json`、Schema 不兼容、锁超时或检测到两个执行者同时推进同一工作项时，停止自动状态推进。保留冲突副本和最后一个完整快照，不以“最后写入者胜出”覆盖事实。

### 10.9 摘要与原始事实冲突

当 `STATE.md`、工作者报告或决策卡与机器事实冲突时，以可验证的原始事实为准，将所有依赖摘要的状态标记为 `stale`，重新生成视图并披露冲突。不得静默修正文案后继续。

### 10.10 项目本质暂时无法确认

用户不知道答案不等于项目永久阻塞。若关键未知可以通过低成本、可撤销实验澄清，苏格拉底之门输出带假设标记的项目本质卡，只允许进入 `experiment_first`，不得伪装成完整方向已经确认。用户确认的是实验目的、成本和停止条件，不是未知假设已经成立。

若核心价值假设无法验证且继续投入会产生重大、难以撤销的后果，保持 `needs_clarification` 或建议停止；不得用更多追问制造虚假的确定感。

## 11. 记忆与自我进化

GuanZai 不把每次成功或失败自动升级为永久规则。项目经验至少记录：

- 适用范围；
- 来源与证据；
- 首次发现和最近验证日期；
- 信心水平；
- 已知反例；
- 复查或退休条件；
- 关联的规则版本。

满足以下任一条件才可提出规则晋升：

- 多次独立任务证明有效；
- 一次严重事故证明必须防范；
- 有明确、可复现的测试证据。

晋升仍需经过审查。经常被合理跳过、与新能力冲突、长期未验证或只增加提示负担的规则应降级或退休。

记忆不直接控制高影响操作。它只能提供候选判断；本次任务仍需重新检查适用范围、当前证据和授权边界。项目、领域和全局规则冲突时，优先使用范围更具体且证据更新的规则；无法判断时不自动选择。

每次路线图复盘应统计：新增规则数量、被合理跳过次数、提示和上下文成本、误路由、过期规则以及被删除或降级的规则。持续进化既包括学习，也包括忘记无效复杂性。

## 12. 测试策略

### 12.1 确定性单元测试

至少覆盖：

- 轻量任务不会被强制分配完整项目流程；
- 明确、低风险的任务跳过苏格拉底之门；
- 方向模糊或高影响任务在本质确认前不能进入 `ready`；
- 本质卡版本变化会使依赖它的方案和验收合同过期；
- 方向未可信的项目推荐 Foolish Think；
- 已批准的直接执行跳过 Foolish Think；
- 能力缺失不会被报告为已调用；
- 安装、可用、执行和验证状态不会混淆；
- 任何代码合并都要求用户批准；
- 合并批准不隐含部署或删除授权；
- 批准对象变化使决策卡失效；
- 高风险行动保留独立验证和恢复要求；
- 机械批量工作不会仅因体量召集团队；
- `STATE.md` 能从相同 `state.json` 确定性重建；
- 状态写入失败不会破坏上一个完整版本；
- 截断和过滤信息会显式呈现；
- 工作者任务包不足时会请求上下文而不是猜测。

### 12.2 契约测试

验证：

- 项目本质卡能够转换为项目定义、任务包和验收合同；
- Foolish Think Stage 1 启用时不会重复运行第二套本质问答；
- Foolish Think 的 `R# -> O# -> S#` 输出能够转换为 GuanZai 验收输入；
- 计划清单字段、状态机和能力状态符合版本化 Schema；
- 不兼容版本会被拒绝或明确降级；
- 决策卡完整包含批准对象和未验证范围；
- 项目思考、任务包、执行结果、证据和决策卡之间的追踪链不丢失；
- 运行、项目、验收和授权状态可以独立表达；
- 模型建议、请求和可观察运行模型不会混淆。

### 12.3 Skill 行为评估

使用成对案例检查：

- 用户提出功能方案但真实问题未知，与已经确认本质的执行任务；
- 能从仓库发现答案的需求，与必须由用户表达偏好的需求；
- 一针见血的单个问题，与没有决策价值的连续盘问；
- AI 通俗复述并请用户纠正，与要求用户证明自己理解；
- 模糊产品想法与明确执行任务；
- 只读安全报告与生产安全修复；
- 小文案修改与架构迁移；
- 测试通过但需求理解错误；
- Foolish Think 已安装、缺失、版本不兼容和当前会话未加载；
- 用户拒绝安装推荐能力；
- 决策卡生成后代码再次变化；
- 短上下文缺少关键限制时的渐进读取；
- 全量历史可用但与当前任务无关时的拒绝加载；
- 工具摘要与原始输出冲突；
- 宿主接受模型请求但不返回实际模型身份。

评估重点是是否做出正确边界判断，不以输出长度或步骤数量作为成功标准。

### 12.4 真实项目试点

在扩大自动化前，选择低风险真实项目验证：

- 用户能否在不阅读代码的情况下正确复述合并影响；
- 建议是否经常被用户无理由推翻；
- 是否出现过“已计划”误报为“已执行”；
- 是否能从保存的恢复点成功撤销；
- 软流程是否经常被合理省略；
- 规则是否减少无脑 Yes/No，而不是增加确认疲劳；
- 新会话恢复是否遗漏关键决定；
- 过滤工具输出是否隐藏过影响结论的信息；
- 每个被用户接受的正确结果所需的总成本是否下降。

### 12.5 路由基准

基准至少比较四组：

| 方案 | 目的 |
| --- | --- |
| 不使用 GuanZai 的宿主默认流程 | 测量 GuanZai 自身开销 |
| 全程高级模型 | 质量上界和成本参考 |
| 全程普通模型 | 低成本基线 |
| GuanZai 自适应路由 | 实验组 |

任务集覆盖简单问答、机械批处理、文档、前端、后端、调试、金融研究和高风险发布。重复运行或使用等价隐藏任务，避免单次偶然结果和任务污染。

主要指标是“每个被接受的正确结果的总成本”，至少包含：

- 输入和输出 token、缓存影响或 WorkBuddy 点数；
- 完成时间和编排启动时间；
- 首次成功率、测试通过率和最终质量盲评；
- 重试次数、升级次数和人工返工；
- 人工检查时间；
- 错误进入正式版本或遗漏关键上下文的损失。

质量未达到预设下限时，不得仅因 token 较少而宣布 Balance 成功。只有具有代表性的任务数据证明总成本下降且质量没有明显恶化，才能修改默认路由。

## 13. 分阶段实施

### 阶段一：State & Context Foundation

这是第一个工程切片，也是后续能力共同依赖的基础：

- 定义版本化 `state.json` Schema、原子写入和并发保护；
- 从机器状态生成短 `STATE.md`；
- 区分私有运行状态和版本化项目知识；
- 建立单决定文件、证据目录和档案索引；
- 工具输出执行预算、脱敏、摘要、截断标记和原始证据保存；
- 定义最小任务包及按需读取协议；
- 测量新会话初始输入、任务总成本、恢复遗漏和过滤误差。

此阶段不启动外部工作者，不改变模型权限，也不声称已经实现项目治理闭环。

### 阶段二：One-Stop Governance

- 更新产品定位、Skill 和架构文档；
- 加入苏格拉底之门、项目本质卡和条件性硬门槛；
- 增加项目状态、风险、硬边界、软建议、证据和批准策略；
- 区分机械、语义和真实效果验收；
- 定义标准合并决策卡和批准对象失效机制；
- 保持所有未执行工作诚实为 `planned`；
- 增加接口契约和边界行为测试。

### 阶段三：可选能力管理

- 建立能力注册表和版本兼容契约；
- 扩展 `guanzai doctor`；
- 增加经同意的一键安装流程；
- 接入 Foolish Think，但保持独立仓库和独立发布；
- 增加缺失、拒绝、失配、未加载和降级测试。

### 阶段四：Observable Execution

- 引入 planned、started、completed、verified、awaiting_approval、approved、merged、failed 和 stale 状态；
- 保存证据、批准对象和决策卡；
- 建立提供商适配器的 detect、capabilities、execute、poll、cancel、normalize 和 usage 契约；
- ACP 可用且经过验证时优先使用，CLI 保持为回退；
- 自动脱敏 ACP 和 CLI 日志中的连接令牌、凭据和隐私字段；
- 区分推荐模型、请求模型和可观察运行模型；
- 在宿主真实支持时组合调用能力，不通过提示词假装执行。

### 阶段五：受控交付

- 管理独立分支和工作区；
- 自动运行项目定义的验证；
- 冻结批准对象并生成决策卡；
- 停在等待用户确认合并；
- 合并之后的部署、清理和不可逆操作继续采用分离授权。

### 阶段六：Benchmark & Adaptive Routing

- 建立覆盖多类真实工作的版本化任务集；
- 比较宿主默认、高级模型、普通模型和 GuanZai 自适应四组；
- 记录成本、时间、质量、重试、升级和人工负担；
- 先设质量下限，再比较每个被接受结果的总成本；
- 只有数据稳定支持时才更新 Balance 默认策略；
- 模型和价格变化只更新能力注册表与评估，不重写治理原则。

每个阶段都必须独立可用，不以未实现的后一阶段来证明当前阶段完整。

## 14. 设计反问与演进闸门

每次新增流程、状态、能力、适配器或记忆规则前，必须回答：

1. **它解决了哪个已观察问题？** 如果只有“可能有用”，先做实验，不进入默认路径；
2. **能否由已有层完成？** 若能，不增加第二套状态、摘要或审批机制；
3. **谁是唯一事实源？** 如果两个组件都能修改同一事实，先重设边界；
4. **上游交给下游什么？** 必须有最小输入、输出、失败和拒绝接收协议；
5. **移除它以后核心还能运行吗？** 专业能力和提供商必须可替换、可缺失、可降级；
6. **未来更强的模型能否合理跳过它？** 方法步骤应保持软约束，授权和证据保持硬边界；
7. **它增加的上下文、延迟和人工负担是多少？** 复杂性必须用结果收益偿还；
8. **失败后如何恢复？** 没有恢复路径的功能不得默认自动运行；
9. **用户是否能理解最终影响？** 无法进入普通语言决策卡的变更，不得进入批准阶段；
10. **这个问题的答案会改变什么？** 如果不会改变方向、风险、边界或验收，就不要询问用户；
11. **什么证据会让我们删除它？** 每项规则都需要复查日期、失败指标或退休条件。

设计评审采用“必要性、单一真相源、可替换性、可恢复性、可理解性、可测量性”六项检查。任一项没有答案时，不把功能加入默认核心。

## 15. 非目标

本次设计不追求：

- 复制 Superpowers 或其他完整开发方法论；
- 把 Foolish Think 复制进 GuanZai 仓库；
- 把苏格拉底之门做成每次任务都强制加载和执行的独立巨型 Skill；
- 对所有任务强制三阶段思考；
- 建设无人工批准的自动合并和自动上线系统；
- 用静态关键词宣称已经理解所有项目语义；
- 根据一次成功自动形成永久记忆；
- 绑定某个角色到永久模型；
- 用更多 Agent 数量作为能力或质量指标；
- 把固定 1–2 KB、固定 worker 数或固定模型名单变成不可调整的产品真理；
- 只为节省 token 而牺牲恢复、授权、证据或最终质量。

## 16. 验收标准

设计实施完成后，应满足：

1. 非程序员能够从 GuanZai 的入口发起想法、研究、内容和代码项目；
2. GuanZai 能说明项目当前状态、风险和为什么选择某条路径；
3. 方向模糊或影响重大的项目通过苏格拉底之门确认本质，明确轻量任务不会被强制盘问；
4. 苏格拉底提问、第一性原理和费曼复述保持软方法，但触发后的本质确认是进入 `ready` 的条件；
5. 项目本质卡变化会精确使受影响的下游方案和验收过期；
6. 模糊方向能够路由到 Foolish Think，且不会与苏格拉底之门重复运行两套需求发现；
7. 只安装 GuanZai 时仍可运行，缺少 Foolish Think 时诚实提示和降级；
8. 用户可以一次同意安装固定、校验过的 Foolish Think 版本；
9. 所有合并请求都附带一屏左右的普通语言决策卡；
10. 用户批准绑定具体不可变对象，变化后必须重新批准；
11. 未经用户批准不合并，合并批准不扩张为部署或不可逆操作授权；
12. 轻量任务保持轻量，高风险任务保留独立验证和恢复能力；
13. 所有状态都能由证据支持，不把计划、检测或自报完成当作验证；
14. 规则可以根据经过验证的经验晋升、降级和退休；
15. 当前未实现的自动执行能力在文档和输出中持续标记为未来阶段；
16. 新会话默认通过短状态视图恢复，并能按需追溯原始决定和证据；
17. 机器事实、语义摘要和长期项目知识不存在未声明的双重真相源；
18. 工具输出被过滤或截断时保留原始证据、范围说明和扩展路径；
19. 子 Agent 不继承完整对话，但不会缺少目标、授权、约束和验收标准；
20. 推荐模型、请求模型和实际可观察模型分别记录；
21. 项目思考、治理计划、执行、验证、决策卡和批准之间可以端到端追踪；
22. GuanZai 的路由收益按每个被接受的正确结果衡量，并包含人工与重试成本；
23. 每个新增核心机制都通过必要性、单一真相源、可替换性、可恢复性、可理解性和可测量性审查；
24. 完整规格和历史不会默认注入每次运行，详细材料只能按当前任务需要渐进加载；
25. 状态更新由实际事件触发，没有变化的普通对话不会重复生成 handoff。

## 17. 最终决定

GuanZai 将发展为一个面向非程序员的一站式 AI 项目保姆，同时保持与模型和提供商无关的小型治理核心。它先通过苏格拉底之门与用户拿下项目本质，再把模糊意图转化为可理解、可验证、可恢复、可持续维护的进展。Foolish Think 作为独立的深层项目思考能力，通过可选依赖、稳定契约、能力检测和经同意的一键安装与 GuanZai 组合。

系统采用“软内核、硬边界、按风险自适应”：允许未来更强的模型自由选择更简洁的工作方法，但不允许它跳过授权、证据、恢复和诚实状态。Context Diet 负责用最小充分上下文保存连续性；能力和模型路由负责选择合适执行者；验证层负责把机器事实连接到项目目标；用户保留重大后果的决定权。

最终生命周期是：

```text
穿过苏格拉底之门，确认项目本质
  -> 必要时用 Foolish Think 深层审视
      -> 建立最小充分上下文
          -> 选择能力
              -> 受控执行
                  -> 证据验证
                      -> 通俗解释
                          -> 用户决定
                              -> 沉淀经过验证的经验
```

代码管理只是项目生命周期的后半段。项目是否值得做、为什么做、执行过程中需要知道什么，以及用什么证据继续投入，必须得到同等重视。

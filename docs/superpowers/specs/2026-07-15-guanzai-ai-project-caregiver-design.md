# GuanZai AI 项目保姆与可组合能力设计

日期：2026-07-15  
状态：已完成对话设计，等待书面审阅

## 1. 背景

GuanZai `v0.3.0-alpha.1` 当前是一个确定性的任务规划核心。它可以判断任务的编排价值、操作影响、审计需要和建议模型，生成标记为 `execution = planned` 的清单，但不会启动工作者、追踪执行、收集验证结果、管理合并或部署。

现有定位主要回答“是否值得委派、由谁处理、使用什么成本层级”。这不足以满足非程序员长期维护项目的需要。用户真正需要的是一个统一入口，帮助其：

1. 在行动前把项目目标、对象、假设和成功标准想清楚；
2. 根据任务状态选择研究、实验、设计、执行、诊断或停止；
3. 在需要代码时保护正式版本，并收集可信的验证证据；
4. 在重要变化前用普通语言解释前因后果；
5. 把最终合并决定保留给用户；
6. 让项目经验能够验证、版本化、更新和退休。

本设计将 GuanZai 从“成本感知的模型路由器”扩展为“AI 项目保姆”，同时保留其小型、可检查和不夸大执行状态的基础。

## 2. 产品定位

### 2.1 用户可见定位

> **GuanZai 是你的 AI 项目保姆。它陪你把事情想清楚，选择合适的能力完成工作，用证据检查结果，并在重要变化发生前帮助你真正看懂和决定。**

“保姆”表示照看、提醒、解释、留痕和保护，不表示替用户拥有项目或替用户接受重大风险。

### 2.2 服务范围

GuanZai 管理的是项目生命周期，不只管理程序代码。项目产物可以是：

- 产品与服务；
- 研究、文章或报告；
- 数据分析和业务规则；
- Agent Skill 或工作流；
- 软件、网站和自动化系统；
- 尚未确定是否值得投入的想法。

代码版本控制、测试、合并和回滚是安全交付能力，只在项目需要代码、配置或可变数据时启用。

### 2.3 用户与 GuanZai 的责任边界

用户负责：

- 提出目标、背景和真实偏好；
- 纠正 GuanZai 对项目的误解；
- 理解剩余影响与风险；
- 决定是否合并正式版本；
- 对上线、删除、付款、授权等高影响操作单独授权。

GuanZai 负责：

- 判断项目当前状态和信息充分程度；
- 选择合适的思考、研究、执行和验证能力；
- 保护稳定成果，保存恢复点；
- 区分计划、执行、验证与批准；
- 用普通语言说明下一步的前因后果；
- 只保留经过验证、仍适用的经验。

## 3. 核心原则

### 3.1 软内核，硬边界

GuanZai 管结果和证据，不强迫所有任务执行相同的思考步骤。

下列内容是可调整的软建议：

- 是否写完整方案；
- 是否使用多个 Agent；
- 是否使用测试驱动开发；
- 是否需要独立审查；
- 是否创建独立工作区；
- 使用哪个模型和推理强度；
- 任务拆成多少步；
- 文档写到什么粒度。

执行者可以省略、合并或替换软步骤，但必须说明省略原因，以及风险如何得到控制。

下列内容是首版不可突破的硬边界：

1. 未经用户批准，不得合并正式版本；
2. 合并批准不等于上线、删除、付款、授权或其他高影响操作的批准；
3. 不得泄露凭据、秘密、个人隐私或受保护数据；
4. 非轻量变更必须存在可用的恢复点；
5. 不得把“测试通过”表述为“绝对没有错误”；
6. 不确定性、未验证范围和失败必须明确披露；
7. 推荐、计划、已执行、已验证和已批准必须使用不同状态；
8. 合并前必须生成标准合并决策卡；
9. 决策卡生成后若批准对象发生变化，原批准失效；
10. 无法用普通语言解释变更时，不得请求用户批准合并。

### 3.2 一站式不等于一体化

用户只需要进入 GuanZai，但 GuanZai 不复制或吞并所有专业能力。专业能力保持独立仓库、独立版本和独立测试，通过稳定的能力契约组合。

### 3.3 最轻量但足够安全

每次任务选择能够控制实际风险的最轻流程。机械工作量、长文本或大量文件本身不构成多 Agent 的理由。只有认知复杂性、独立性价值或行动后果足以覆盖编排成本时才委派。

### 3.4 证明结果，不规定思维表演

GuanZai 不要求模型展示固定的思维步骤。它要求可检查的外部产物：项目定义、证据、差异、测试结果、恢复点、决策卡和状态记录。

## 4. 总体架构

```text
用户想法、问题或项目
          |
          v
GuanZai 入口：理解处境、权限与项目状态
          |
          +--> 方向未可信 --> 项目思考能力（Foolish Think，可选依赖）
          |                         |
          |                         v
          |                 经审视的项目定义与验证路线图
          |
          +--> 方向已确认 --> 研究 / 设计 / 执行 / 诊断
                                    |
                                    v
                            证据与可恢复成果
                                    |
                                    v
                         标准决策卡与用户批准节点
                                    |
                                    v
                         合并；其他高影响操作另行授权
```

架构分为四层：

1. **治理层**：GuanZai 判断状态、风险、所需能力、授权和证据；
2. **能力层**：Foolish Think、研究、设计、开发、测试、审计等独立能力；
3. **执行层**：Codex、WorkBuddy、其他 Agent、CLI 和人工执行者；
4. **项目状态层**：计划、证据、恢复点、决策卡、批准和经过验证的记忆。

新增能力不能改变硬边界的含义。提供商、模型和 Skill 属于能力或执行层，不拥有合并授权。

## 5. Foolish Think 的组合方式

### 5.1 架构决定

Foolish Think 与 GuanZai 保持两个仓库、两个 Skill 和独立发布。GuanZai 不复制 Foolish Think 的完整指令，只保留：

- 能力标识与用途；
- 语义触发条件；
- 输入输出契约；
- 支持版本范围；
- 官方仓库和安装来源；
- 能力检测、缺失提示与降级规则。

Foolish Think 官方仓库：<https://github.com/Dovesoup/foolish-think>

### 5.2 语义路由

当项目定义、核心假设、可行性、方向或实际结果尚不可信时，GuanZai 推荐项目思考能力：

- `Seed`：只有需求或早期想法；
- `Stress-test`：已有方案但尚未发生重大失败；
- `Diagnose`：执行结果与预期存在重要差距。

下列情况默认跳过 Foolish Think：

- 已批准方案下的直接执行；
- 独立且可复现的技术故障；
- 普通知识问答；
- 不需要战略审视的轻量修改；
- 用户明确要求不做方向挑战，且行动不触发硬安全边界。

当另一个 brainstorming 流程已在运行时，不重复启动第二套需求发现对话。保留当前对话和批准节奏，同时应用 Foolish Think 的假设审计、失败预演、可追溯性和证伪标准。

### 5.3 交接契约

Foolish Think 的项目级输出向 GuanZai 提供三类可追溯对象：

```text
R#：事实、假设、风险或根因发现
  -> O#：针对具体发现的方向调整
      -> S#：能够验证或证伪方向的行动单元
```

`S#` 成为 GuanZai 后续计划的验收输入，至少包含：

- 目标和交付物；
- 输入与输出契约；
- 客观验收证据；
- 最小证伪实验；
- 失败成本与可逆性；
- 依赖和下游消费者；
- `do now`、`experiment first` 或 `restart` 决定。

这样，项目最初的关键假设能够追踪到最终验收和合并决策，不允许执行者在末端自行改写成功标准。

### 5.4 安装与依赖管理

GuanZai 核心必须能够独立安装和运行。Foolish Think 是推荐但可选的能力依赖。

未来的 `guanzai setup` 提供一站式安装：

1. 显示核心和推荐能力；
2. 解释每项能力的用途、来源、版本和数据边界；
3. 获得用户明确同意后，从官方仓库获取固定版本；
4. 校验版本、发布来源和文件完整性；
5. 安装到当前 Agent 平台的个人 Skill 目录；
6. 检查文件是否存在，并区分“已安装”和“当前会话已加载”；
7. 平台需要重启时明确提示；
8. 记录已安装版本和兼容性结果。

不得静默下载 `latest`，不得自动执行未经固定和校验的远程安装脚本。

未来的 `guanzai doctor` 对每项能力报告：

- `missing`：未安装；
- `installed`：文件存在，但当前会话是否加载未知；
- `available`：当前宿主明确暴露该能力；
- `incompatible`：版本不在支持范围；
- `unverified`：只能检测文件，无法证明运行可用。

当任务需要 Foolish Think 但能力不可用时，GuanZai 必须：

1. 说明为什么建议使用；
2. 说明当前缺失状态；
3. 提供经用户确认的一键安装；
4. 允许用户拒绝；
5. 拒绝后使用基础澄清能力继续，并披露未执行完整假设审查；
6. 不得模仿或声称已经运行完整 Foolish Think。

## 6. 项目生命周期

### 6.1 入口状态判断

每项工作先归入一个主状态：

- `idea`：想法或需求尚未形成可信项目；
- `proposal`：已有方案，尚待压力测试或批准；
- `ready`：方向和验收条件已确认，可以执行；
- `active`：项目正在执行；
- `blocked`：执行受阻，但根因未明；
- `degraded`：已有成果，但实际效果不符合预期；
- `complete`：证据满足验收，等待或已经完成相应批准；
- `stopped`：决定停止投入，并记录原因与保留成果。

状态判断可以由强模型进行语义分析。确定性规则只作为可解释的保底，不应假装具有完整语义理解。

### 6.2 风险等级

| 等级 | 典型任务 | 默认路径 |
| --- | --- | --- |
| 轻量 | 文案、格式、局部样式、可轻易撤销的整理 | 直接执行、结果检查、简短说明 |
| 普通 | 新功能、数据处理、规则或工作流修改 | 简短方案、隔离执行、验证、标准决策卡 |
| 高风险 | 生产数据、权限、安全、财务、架构迁移、不可逆操作 | 项目/方案审查、独立验证、恢复演练、标准决策卡和分离授权 |

风险取决于行动和影响，不取决于敏感词本身。只读安全说明不等同于安全系统变更；一行生产计费公式修改仍是高风险任务。

### 6.3 代码与可变项目的安全交付

普通及高风险代码变更默认：

```text
已确认的正式版本
  -> 独立分支或工作区
      -> 小步修改与可识别恢复点
          -> 自动测试和必要的真实效果检查
              -> 独立验证（按风险）
                  -> 标准合并决策卡
                      -> 等待用户批准合并
```

用户选择的默认授权边界是：**自动工作只能运行到“等待确认合并”为止。**

以下操作不包含在合并批准中，必须根据影响单独请求授权：

- 部署到生产环境；
- 数据删除或不可逆迁移；
- 权限、密钥和访问控制变更；
- 付款、转账或计费；
- 清理仍承担恢复用途的分支、工作区、备份或临时数据库。

若项目没有 Git，GuanZai 应优先创建可恢复副本或快照，并明确说明保护程度低于完整版本控制，不得使用“合并”掩盖普通文件覆盖。

## 7. 标准合并决策卡

决策卡默认控制在一屏左右，以普通语言回答：用户批准后世界会发生什么变化。

```text
结论：建议合并 / 建议暂缓 / 需要你先检查

原本的问题：
这次做了什么：
合并后的实际变化：
为什么采用这个方案：
验证了哪些情况：
没有验证或仍不确定的地方：
可能影响谁：
出现问题如何恢复：
AI 的明确建议：
你现在批准的具体事项：
批准对象指纹：
```

### 7.1 生成规则

- 不得只报告测试数量、覆盖率或 CI 颜色；
- 必须解释验证证据证明了什么、没有证明什么；
- 必须区分程序正确性、需求正确性和真实使用效果；
- 必须披露未运行、失败、跳过或不适用的验证；
- 必须给出明确建议和依据，不能把判断完全推回给用户；
- 技术附件可以保留，但不得替代普通语言摘要。

### 7.2 批准对象

决策卡必须绑定不可歧义的批准对象，例如 Git 提交 SHA、不可变构建标识或文件快照哈希。

生成决策卡后，只要批准对象发生变化：

1. 原决策卡标记为 `stale`；
2. 原批准自动失效；
3. 重新验证变化部分及其影响；
4. 生成新决策卡；
5. 再次等待用户批准。

## 8. 状态与证据模型

### 8.1 状态必须真实

建议使用下列工作状态：

- `recommended`：建议使用某能力；
- `planned`：已生成计划；
- `available`：宿主暴露能力；
- `started`：执行已经开始；
- `completed`：执行者报告完成；
- `verified`：验收证据经过检查；
- `awaiting_approval`：批准对象已冻结并生成决策卡；
- `approved`：用户批准指定对象；
- `merged`：指定对象已经进入正式版本；
- `failed`：未完成或证据不满足；
- `stale`：对象变化导致计划、证据或批准失效；
- `stopped`：基于明确决定停止投入。

状态只能由相应证据推进。例如，存在 CLI 不代表 `available`，生成命令不代表 `started`，执行者自报完成不代表 `verified`。

### 8.2 计划清单扩展

后续清单应增加：

```json
{
  "project_state": "proposal",
  "risk_level": "ordinary",
  "hard_boundaries": [],
  "soft_playbook": [],
  "required_evidence": [],
  "approval_policy": {
    "merge": "user_required",
    "deployment": "separate_authorization"
  },
  "required_capabilities": [
    {
      "name": "project-thinking",
      "provider": "foolish-think",
      "status": "missing",
      "required": false,
      "reason": "项目方向和关键假设尚未得到验证"
    }
  ],
  "decision_card_required": true
}
```

字段是机器契约，面向用户的输出仍需转换为普通语言。

## 9. 失败与降级处理

### 9.1 信息不足

如果缺失信息可能改变项目方向、根因判断或第一项不可逆决定，停止扩展方案，每次只询问一个最高信息价值问题。

### 9.2 能力缺失

说明缺少什么、为什么需要、如何安装、拒绝后的影响。降级继续时必须标记哪些检查未执行。

### 9.3 无可靠验证方法

不得用“看起来正常”替代证据。提供最低成本的替代验证，或将结论改为“需要你先检查”。

### 9.4 工作区不安全

遇到用户未提交的修改、来源不明的文件、无法隔离的正式环境或没有恢复点时，不覆盖、不清理；先保护现状，再说明阻碍。

### 9.5 验证失败

保持正式版本不变。修复后重新运行受影响验证；连续失败应重新检查方案假设，而不是无限重复同一路径。

### 9.6 批准过期

批准对象变化后不沿用旧的 Yes。状态退回 `stale`，重新生成证据和决策卡。

### 9.7 无法恢复

在第一次难以撤销的操作前停止，说明最坏损失和替代方案，获得单独授权；不能把它包装为普通合并。

## 10. 记忆与自我进化

GuanZai 不把每次成功或失败自动升级为永久规则。项目经验至少记录：

- 适用范围；
- 来源与证据；
- 首次发现和最近验证日期；
- 信心水平；
- 已知反例；
- 复查或退休条件；
- 关联的规则版本。

满足以下任一条件才可提出规则晋升：

- 多次独立任务证明有效；
- 一次严重事故证明必须防范；
- 有明确、可复现的测试证据。

晋升仍需经过审查。经常被合理跳过、与新能力冲突、长期未验证或只增加提示负担的规则应降级或退休。

## 11. 测试策略

### 11.1 确定性单元测试

至少覆盖：

- 轻量任务不会被强制分配完整项目流程；
- 方向未可信的项目推荐 Foolish Think；
- 已批准的直接执行跳过 Foolish Think；
- 能力缺失不会被报告为已调用；
- 安装、可用、执行和验证状态不会混淆；
- 任何代码合并都要求用户批准；
- 合并批准不隐含部署或删除授权；
- 批准对象变化使决策卡失效；
- 高风险行动保留独立验证和恢复要求；
- 机械批量工作不会仅因体量召集团队。

### 11.2 契约测试

验证：

- Foolish Think 的 `R# -> O# -> S#` 输出能够转换为 GuanZai 验收输入；
- 计划清单字段、状态机和能力状态符合版本化 Schema；
- 不兼容版本会被拒绝或明确降级；
- 决策卡完整包含批准对象和未验证范围。

### 11.3 Skill 行为评估

使用成对案例检查：

- 模糊产品想法与明确执行任务；
- 只读安全报告与生产安全修复；
- 小文案修改与架构迁移；
- 测试通过但需求理解错误；
- Foolish Think 已安装、缺失、版本不兼容和当前会话未加载；
- 用户拒绝安装推荐能力；
- 决策卡生成后代码再次变化。

评估重点是是否做出正确边界判断，不以输出长度或步骤数量作为成功标准。

### 11.4 真实项目试点

在扩大自动化前，选择低风险真实项目验证：

- 用户能否在不阅读代码的情况下正确复述合并影响；
- 建议是否经常被用户无理由推翻；
- 是否出现过“已计划”误报为“已执行”；
- 是否能从保存的恢复点成功撤销；
- 软流程是否经常被合理省略；
- 规则是否减少无脑 Yes/No，而不是增加确认疲劳。

## 12. 分阶段实施

### 阶段一：治理契约

- 更新产品定位、Skill 和架构文档；
- 增加项目状态、风险、硬边界、软建议、证据和批准策略；
- 定义标准合并决策卡；
- 保持所有执行状态诚实为 `planned`；
- 增加相应测试。

### 阶段二：可选能力管理

- 建立能力注册表和版本兼容契约；
- 扩展 `guanzai doctor`；
- 增加经同意的一键安装流程；
- 接入 Foolish Think，但保持独立仓库和独立发布；
- 增加缺失、拒绝、失配和未加载测试。

### 阶段三：持久执行账本

- 引入 planned、started、completed、verified、awaiting_approval、approved、merged、failed 和 stale 状态；
- 保存证据、批准对象和决策卡；
- 支持轮询、取消和结果规范化；
- 在宿主真实支持时组合调用能力，不通过提示词假装执行。

### 阶段四：受控交付

- 管理独立分支和工作区；
- 自动运行项目定义的验证；
- 冻结批准对象并生成决策卡；
- 停在等待用户确认合并；
- 合并之后的部署、清理和不可逆操作继续采用分离授权。

每个阶段都必须独立可用，不以未实现的后一阶段来证明当前阶段完整。

## 13. 非目标

本次设计不追求：

- 复制 Superpowers 或其他完整开发方法论；
- 把 Foolish Think 复制进 GuanZai 仓库；
- 对所有任务强制三阶段思考；
- 建设无人工批准的自动合并和自动上线系统；
- 用静态关键词宣称已经理解所有项目语义；
- 根据一次成功自动形成永久记忆；
- 绑定某个角色到永久模型；
- 用更多 Agent 数量作为能力或质量指标。

## 14. 验收标准

设计实施完成后，应满足：

1. 非程序员能够从 GuanZai 的入口发起想法、研究、内容和代码项目；
2. GuanZai 能说明项目当前状态、风险和为什么选择某条路径；
3. 模糊方向能够路由到 Foolish Think，明确执行任务不会被强制三思；
4. 只安装 GuanZai 时仍可运行，缺少 Foolish Think 时诚实提示和降级；
5. 用户可以一次同意安装固定、校验过的 Foolish Think 版本；
6. 所有合并请求都附带一屏左右的普通语言决策卡；
7. 用户批准绑定具体不可变对象，变化后必须重新批准；
8. 未经用户批准不合并，合并批准不扩张为部署或不可逆操作授权；
9. 轻量任务保持轻量，高风险任务保留独立验证和恢复能力；
10. 所有状态都能由证据支持，不把计划、检测或自报完成当作验证；
11. 规则可以根据经过验证的经验晋升、降级和退休；
12. 当前未实现的自动执行能力在文档和输出中持续标记为未来阶段。

## 15. 最终决定

GuanZai 将发展为一个面向非程序员的一站式 AI 项目保姆，但保持小型治理核心。Foolish Think 作为独立的项目思考能力，通过可选依赖、稳定契约、能力检测和经同意的一键安装与 GuanZai 组合。

系统采用“软内核、硬边界、按风险自适应”：允许未来更强的模型自由选择更简洁的工作方法，但不允许它跳过授权、证据、恢复和诚实状态。代码管理只是项目生命周期的后半段；项目是否值得做、为什么做以及用什么证据继续投入，必须在执行之前得到同等重视。
